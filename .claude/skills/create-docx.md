# Skill: /create-docx

> Генерация профессиональных .docx документов (брифинги, КП, аналитика) по твоей дизайн-системе

## Триггеры

- `/create-docx`
- "создай docx"
- "сделай брифинг в docx"
- "сгенерируй документ"
- "подготовь docx для клиента"

## Дизайн-система

**ОБЯЗАТЕЛЬНО прочитай перед началом работы** — положи свою дизайн-систему по пути:

```
docs/docx-design-system.md
```

Файл с дизайн-системой должен содержать:
- Цветовую палитру (`C.*`)
- Библиотеку компонентов (хелперы: `R`, `P`, `sp`, `bullet`, `secH1`, `secH2`, `accentBox`, `twoColBox`, `solutionCard`, `painBlock`, `TC`, `THC`, `TR`)
- Настройку документа (numbering, styles, headers/footers)
- Стандартную структуру брифинга
- Типовые паттерны (сниппеты)
- Критические правила (что нельзя делать)

---

## Алгоритм работы

1. **Прочитай** `docs/docx-design-system.md` полностью
2. **Прочитай** входной файл (материал для документа)
3. **Создай** Node.js скрипт:
   - Скопируй ВЕСЬ блок хелперов из секции 4 дизайн-системы
   - Используй стандартную структуру из секции 5
   - Адаптируй header/footer под конкретного клиента и тип документа
   - Заполни `children: [...]` данными из входного файла
4. **Убедись**, что `output/` существует: `mkdir -p output`
5. **Установи** зависимость: `npm list -g docx || npm install -g docx`
6. **Запусти** скрипт: `node generate-doc.js`
7. **Валидируй** результат (если есть скрипт валидации)
8. **Удали** временный скрипт после успешной генерации

---

## Критические правила (кратко)

- **НЕ** использовать символ `•` напрямую — только через `numbering`
- **НЕ** использовать `\n` внутри TextRun — только отдельные Paragraph
- **НЕ** использовать `WidthType.PERCENTAGE` — ломается в Google Docs
- **НЕ** использовать `ShadingType.SOLID` — даёт чёрный фон
- **ВСЕГДА** `margins` на ячейках таблиц
- **ВСЕГДА** двойная ширина: `columnWidths` + `width` на каждой ячейке
- **ТОЛЬКО** цвета из палитры `C.*`
- **ТОЛЬКО** компоненты из дизайн-системы — не изобретать новые
