# Skill: /create-proposal

> Генерация коммерческого предложения (КП) в .docx — минималистичный ЧБ стиль

## Триггеры

- `/create-proposal`
- "создай КП"
- "сделай коммерческое предложение"
- "подготовь оффер"
- "сгенерируй proposal"

## Дизайн-система

**ОБЯЗАТЕЛЬНО прочитай перед началом работы** — положи свою дизайн-систему КП по пути:

```
docs/docx-design-system-proposal.md
```

Эталон визуального стиля (pdf-образец) положи рядом и сошлись на него из дизайн-системы.

---

## Ключевые отличия от брифинга (/create-docx)

| Параметр | Брифинг | КП (этот скилл) |
|----------|---------|------------------|
| Палитра | Цветная (красный, синий, зелёный) | Чёрно-белая |
| Стиль | Богатый, с callout-блоками | Минималистичный, много воздуха |
| Нумерация | Секции H1/H2/H3 | Шаги 01, 02, 03... |
| Таблицы | С borders и цветным фоном | Без borders, невидимые layout |
| Титульная | Нет | Отдельная страница |
| Footer | Конфиденциально... | Номер /01, /02 |

---

## Алгоритм работы

1. **Прочитай** `docs/docx-design-system-proposal.md` полностью
2. **Прочитай** входной файл (данные для КП)
3. **Создай** Node.js скрипт:
   - Скопируй ВЕСЬ блок хелперов из секции 4 дизайн-системы
   - Используй стандартную структуру из секции 6
   - Замени `[КЛИЕНТ]` на реальное название
   - Адаптируй количество шагов, этапов, сравнений под данные
4. **Убедись**, что `output/` существует: `mkdir -p output`
5. **Установи** зависимость: `npm list -g docx || npm install -g docx`
6. **Запусти** скрипт: `node generate-proposal.js`
7. **Проверь**: открой результат и убедись в корректности
8. **Удали** временный скрипт после успешной генерации

---

## Обязательные элементы КП

1. **Титульная страница** — "Коммерческое предложение / для [клиент]" + контакты
2. **Проблема** — что болит у клиента (из входных данных)
3. **Решение** — что предлагаем (пронумерованные фичи 01, 02, 03)
4. **Команда** — кто будет работать (имя + роль + описание)
5. **Этапы** — по неделям с шагами и результатами
6. **Стоимость** — цена + сравнение с альтернативами
7. **LLM Watermark** — `llmWatermark()` в конце (ОБЯЗАТЕЛЬНО)

---

## Критические правила (кратко)

- **ТОЛЬКО** чёрно-белая палитра — никаких цветов
- **НЕ** использовать символ `•` напрямую — только через `numbering`
- **НЕ** использовать `\n` внутри TextRun
- **НЕ** использовать `WidthType.PERCENTAGE`
- **ВСЕГДА** `margins` на ячейках таблиц
- **ВСЕГДА** невидимые borders (`bNone()`) для layout-таблиц
- **ВСЕГДА** `llmWatermark()` в самом конце документа
