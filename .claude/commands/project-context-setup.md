# Project Context Setup

Ты помогаешь создать базовый контекст для нового проекта. По методике Ильи Мартына, каждый проект должен иметь 4 фундаментальных файла, которые радикально улучшают качество работы AI внутри этого проекта.

> "Это 4 базовых файла, которые должен иметь каждый проект. Даже если вы больше ничего не делаете, просто общение внутри этого контекста будет давать значительно лучшие результаты."

---

## ПРОЦЕСС

### Шаг 1: Сбор информации

Прочитай $ARGUMENTS — там может быть:
- Название проекта
- Краткое описание
- Или просто тема/идея

Задай пользователю вопросы по каждому из 4 файлов (одним блоком, не по одному):

**Про проект:**
- Что это за продукт/сервис? В одном предложении.
- Какую проблему решает?
- Текущая стадия (идея / MVP / работающий продукт)?
- Ключевые метрики или цели?

**Про аудиторию:**
- Кто главный пользователь? (роль, размер компании, контекст)
- Зачем им это? Какая боль?
- Как они решают эту проблему сейчас (без вашего продукта)?
- Что для них "вау"?

**Про конкурентов:**
- Кто главные 3-5 конкурентов (или альтернатив)?
- Чем ваш подход отличается?
- Почему клиент выберет вас, а не их?

**Про тон коммуникации:**
- Формальный или неформальный?
- Ключевые слова/термины, которые используете?
- Чего точно НЕ говорите? (антипаттерны)
- Примеры коммуникации, которые нравятся (бренды, авторы)?

### Шаг 2: Определи место

Выбери папку для проекта:
- Клиентский/рабочий проект → в рабочем подпроекте (например, `<work>/CRM/Active/[клиент]/`)
- Личный проект → в личном подпроекте (например, `<personal>/Projects/`)
- Новый проект — создай папку в подходящем месте

### Шаг 3: Создай 4 файла

Создай файлы в папке проекта:

---

#### Файл 1: `project-description.md`

```markdown
# [Название проекта]

## Что это
[1-2 предложения — суть продукта]

## Проблема
[Какую проблему решает, для кого]

## Решение
[Как решает, ключевой механизм]

## Стадия
[Идея / MVP / Работающий продукт / Масштабирование]

## Ключевые метрики
- [метрика 1]: [текущее значение → целевое]
- [метрика 2]: [текущее значение → целевое]

## Tech Stack (если применимо)
- [технология]: [зачем]

## Ключевые ограничения
- [ограничение 1]
- [ограничение 2]
```

---

#### Файл 2: `audience.md`

```markdown
# Аудитория: [Название проекта]

## Основной сегмент

**Кто:** [роль, размер компании, контекст]
**Боль:** [главная проблема, которую решаем]
**Текущее решение:** [как живут без нас]
**Момент осознания:** [когда понимают, что нужен наш продукт]
**"Вау" момент:** [что заставляет сказать "беру!"]

## Jobs to Be Done
1. Когда я [ситуация], я хочу [действие], чтобы [результат]
2. ...

## Что важно для них
- [приоритет 1]
- [приоритет 2]

## Что их отталкивает
- [антипаттерн 1]
- [антипаттерн 2]

## Цитаты / голос клиента
> "[реальная или типичная цитата]"
```

---

#### Файл 3: `competitors.md`

```markdown
# Конкуренты: [Название проекта]

## Ландшафт

| Конкурент | Тип | Сильные стороны | Слабые стороны | Цена |
|-----------|-----|----------------|----------------|------|
| [имя] | [прямой/косвенный/альтернатива] | ... | ... | ... |

## Наше отличие

**Positioning statement:**
Для [аудитория], которые [проблема], [продукт] — это [категория], которая [ключевое отличие]. В отличие от [главный конкурент], мы [главное преимущество].

## Чему стоит научиться у конкурентов
- У [конкурент 1]: [что делают хорошо]
- У [конкурент 2]: [что делают хорошо]

## Чего НЕ делать как конкуренты
- [антипаттерн от конкурента]
```

---

#### Файл 4: `tone-of-voice.md`

```markdown
# Tone of Voice: [Название проекта]

## Характер бренда
[3-5 прилагательных, описывающих тон]

## Мы звучим как
- [описание стиля]
- [пример фразы в нашем стиле]

## Мы НЕ звучим как
- [антипаттерн]
- [пример фразы НЕ в нашем стиле]

## Ключевые термины
| Мы говорим | Мы НЕ говорим |
|-----------|---------------|
| [термин] | [антитермин] |

## Примеры (reference)
- [бренд/автор, стиль которого близок]

## Правила
1. [правило коммуникации 1]
2. [правило коммуникации 2]
```

### Шаг 4: Подтверди

Покажи пользователю краткое резюме всех 4 файлов. Спроси, нужны ли корректировки.

---

## ВАЖНЫЕ ПРАВИЛА

1. **Задавай вопросы одним блоком**, не по одному — чтобы пользователь мог ответить за раз.
2. **Если пользователь не знает ответ** — предложи варианты на основе описания проекта.
3. **Не перегружай файлы.** Лучше короткий и точный, чем длинный и размытый.
4. **Используй язык пользователя.** Если он говорит "юзеры" — не пиши "пользователи".
5. **Файлы должны быть полезны для AI.** Цель — чтобы любой AI-инструмент, читающий эти файлы, давал значительно более точные ответы по проекту.
