# Prospect Brief

Ты генерируешь **внешний документ** для партнёра или клиента на основе уже готового внутреннего анализа (`business-analysis.md`). Этот документ можно отправить за пределы команды.

---

## ВХОДНЫЕ ДАННЫЕ

Из $ARGUMENTS извлеки:
- **Название компании** (обязательно)
- **Для кого документ** (опционально): имя партнёра, его роль, контекст встречи
- **Дата встречи** (опционально): если известна

Если название не указано — спроси.

---

## ПРОЦЕСС

### Шаг 1: Найти внутренний анализ

1. Поищи папку компании в `<твой-проект>/CRM/Active/` (Glob по имени компании)
2. Прочитай `business-analysis.md` — это основа для брифинга
3. Если есть `deal.md` — прочитай тоже (партнёр, контакт, структура сделки, следующие шаги)
4. Если `business-analysis.md` не найден — скажи пользователю: «Сначала запусти `/prospect-research` для этой компании»

### Шаг 2: Собрать брифинг

Из внутреннего анализа **извлеки и переформатируй** в чистый документ:

**Включить:**
- Профиль компании (таблица, без ИНН/ОГРН — это внутренние данные)
- Структура бизнеса (tree-диаграмма, если есть)
- Что происходит с бизнесом (стратегический контекст)
- Основные проблемы (3-4 ключевые, с оцифровкой потерь) — **без** оценок продаваемости
- Наши предложения: входной продукт → основное решение → апсейл
- Что можно закрыть готовыми инструментами (показывает экспертизу и честность)
- Почему мы (твоя компания)
- Вопросы для первой встречи
- Следующие шаги (из deal.md, если есть)

**НЕ включать:**
- Оценки продаваемости (Боль/Стоимость/Продаваемость — это внутреннее)
- Формулировки «клиент может сделать сам» / «риск замены»
- Вопросы про бюджет (слишком рано для первой встречи через посредника)
- Наши сомнения и внутренние комментарии
- ИНН, ОГРН, юридические данные (если только партнёр не юрист)

### Шаг 3: Адаптировать под контекст

Если указан партнёр/контекст:
- Упомяни его роль в «Следующие шаги»
- Адаптируй формулировки: если партнёр — технический, добавь больше деталей по решениям; если бизнес — больше про ROI и стоимость проблем
- Вопросы для встречи: убери те, что не релевантны для этого контакта

Если не указан — сделай универсальный документ.

### Шаг 4: Сохранить

Сохрани как `briefing-for-partner.md` (или `briefing-for-[имя].md` если указан конкретный партнёр) в ту же папку, где лежит `business-analysis.md`.

Выдай в чат краткое резюме: для кого документ, основные блоки, путь к файлу.

## ВАЖНЫЕ ПРАВИЛА

1. **Не выдумывай.** Бери данные только из `business-analysis.md` и `deal.md`. Не добавляй информацию, которой нет в анализе.
2. **Чистый язык.** Никаких «мы считаем, что клиент может...» — это внутренние мысли.
3. **Уверенный тон.** Пиши как эксперт, а не как тот, кто сомневается. Но без агрессии.
4. **Write-first.** Результат в файл, в чат — краткое резюме.
5. **Сохраняй рядом.** Брифинг лежит в той же папке, что и анализ.
