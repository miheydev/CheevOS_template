# Multi-Project Workspace Template

> Корневая папка multi-project workspace для работы с Claude Code. Каждая подпапка — отдельный проект со своими правилами.
>
> **Что это:** шаблон-скелет со скиллами, командами и базовыми правилами. Клонируй, переименуй, наполняй проектами.

## Quick Start

**Все проекты, статусы и ключевые файлы** → [PROJECTS.md](./PROJECTS.md)
**Запуск сервисов** → [SERVICES.md](./SERVICES.md)

## Контекстная изоляция

**Перед любым действием определи, в каком проекте работаешь.**

Каждый подпроект может иметь собственный `CLAUDE.md` со своими правилами. При работе в конкретном проекте Claude Code автоматически загрузит его `CLAUDE.md`.

Пример:
- `<project-1>/CLAUDE.md` — правила проекта 1
- `<project-2>/CLAUDE.md` — правила проекта 2

**НИКОГДА** не делай git-операции, затрагивающие несколько проектов одновременно.

## Формат ссылок между файлами

**Используй стандартные markdown-ссылки с относительными путями** (работают в VS Code):

```markdown
[название](./подпапка/файл.md)           ← внутри одного проекта
[название](../другой-проект/файл.md)     ← между проектами
```

**НЕ используй** Obsidian wikilinks `[[path/file]]` — они не кликабельны в VS Code preview.

## INBOX

Корневой `INBOX/` — **временная** папка для файлов, которые нужно распределить по проектам.

Скажи "разбери INBOX" → AI прочитает каждый файл и переместит в нужный проект.

**НЕ проверяй INBOX автоматически при старте сессии** — это тратит контекст. Заходи в INBOX только когда пользователь попросит.

### Критическое правило: после работы с файлом из INBOX — убери его оттуда

**Если взял файл из INBOX для работы — после обработки перемести его в нужное место:**
1. Определи, к какому проекту относится файл
2. **Перемести** (не копируй) в целевую папку проекта
3. Обнови связанную документацию (если применимо)

**Никогда не кладь обработанный файл обратно в INBOX.** Это временная папка, не архив.

Подробный воркфлоу → [.claude/skills/inbox-root-processing.md](./.claude/skills/inbox-root-processing.md)

## Критические правила (ОБЯЗАТЕЛЬНО)

### 0. Git-коммиты без Co-Authored-By

**НЕ добавляй** строку `Co-Authored-By` в коммит-сообщения. Никаких автоматических упоминаний AI-ассистентов в коммитах, если пользователь явно не попросил.

### 1. Читай перед изменением

**НИКОГДА** не перезаписывай файл без предварительного чтения.
- Сначала `Read` → понять что там → потом `Edit` или `Write`
- Исключение: создание нового файла

### 2. Создание артефактов: Write-First

**Принцип:** Если результат работы ценнее одной сессии — пиши в файл, не держи в чате.

**Создавай БЕЗ вопросов:**
- Оценка проекта → полный набор артефактов (intake, wbs, estimate, etc.)
- Работа со сделкой → deal.md + changelog.md
- Обработка встречи → саммари + обновление CRM
- Анализ/исследование → результат в соответствующую папку базы знаний

**НЕ создавай:**
- Обёртки, утилиты, абстракции "на будущее"
- Документацию к коду, который не менялся
- Файлы-дубли существующих

**Правило scratchpad:** Если анализ занимает >1 экрана текста — сначала запиши результат в файл, потом дай краткое резюме в чат.

### 3. Безопасность

- **НЕ** коммить `.env`, credentials, API-ключи
- **НЕ** выводи секреты в output
- **НЕ** копируй персональные данные между проектами
- Проверяй `.gitignore` перед коммитом
- При обнаружении секрета в коде → немедленно сообщи
- **ВСЕГДА** пинь версии npm-пакетов в `.mcp.json`. НЕ использовать `@latest` или пакеты без версии в `npx`-командах. Причина: supply chain — компрометация npm-пакета даёт RCE через MCP-сервер при каждом запуске.
- **НЕ** храни токены в `env`-блоке `.mcp.json`. Используй `.env` + `dotenv` внутри сервера. `.mcp.json` должен быть в `.gitignore`, т.к. содержит локальные пути и может быть случайно заполнен секретами.

### 4. Генерация текстов: без AI-штампов

При генерации любого текста от имени автора (посты, письма, КП, сообщения):
- Сначала прочитай свой ToV-профиль, если есть (пример пути: `brand/tov-profiles/`)
- Используй скилл `/tov-write` или `/mega-polish` для полировки
- **НЕ** используй: negative parallelism ("не X, а Y"), риторические самопостановки ("Результат? Разрушительный."), вводные-пустышки ("стоит отметить"), канцелярит
- Сохраняй разговорные обороты автора — это НЕ филлеры, а живая речь
- Чек-лист маркеров → [.claude/skills/ai-markers.md](./.claude/skills/ai-markers.md)

### 5. Date Anchor — перед записью даты в файл

Перед тем как записать дату в имя файла, имя артефакта или внутрь плана — произнеси якорь:
`Anchor: today = YYYY-MM-DD (DOW)` из системного хука.

- **НИКОГДА** не вычисляй день недели из даты сам — высокий риск ошибки
- **НИКОГДА** не якорь по датам из окружающих файлов — они могут быть устаревшими
- Источник истины — текущий контекст сессии

### 6. Никаких секретов в чат

**НИКОГДА** не печатай в чат API-ключи, токены, client_secret, session string, пароли — даже если они видны в контексте файла. Чат может быть залогирован или расшарен.

При grep по секретам используй `grep -l` (только имена файлов) или присваивай в переменную — НЕ `grep -h` / `head`, которые выведут значение в stdout.

### 7. Self-check перед "готово"

Для задач с >1 пунктом — перед финальным "готово" пройдись по списку запрошенного и отметь статус каждого пункта. Только потом давай саммари в чат.

- Список запрошенного → статус (done / skipped / unclear) → доделать пропущенное → саммари
- Для односоставных задач (один файл, одна команда) — не нужно, overkill

## Добавление новых правил

| Тип правила | Куда добавлять |
|-------------|----------------|
| **Критическое / автоматическое** (должно работать ВСЕГДА) | Прямо в этот файл (`CLAUDE.md`) |
| **Проектное** (для конкретного проекта) | `<project>/CLAUDE.md` |
| **On-demand** (вызывается по запросу) | `.claude/skills/` или `.claude/commands/` |
| **Урок / паттерн / предпочтение** (контекстное, может устареть) | Memory (feedback или project тип) |

*CLAUDE.md — жёсткие правила, которые работают всегда. Memory — контекстные знания, которые могут меняться.*

### Формат правил

- Начинай с директивы: **НИКОГДА** / **ВСЕГДА** / **НЕ**
- Сначала почему, потом что делать (1-2 строки)
- Пример кода/команды — только если без него неясно
- Никаких абзацев — только буллеты
- Одно правило = одна конкретная ситуация (не "будь аккуратнее")

### Anti-bloat

- **НЕ** добавляй правило, если оно дублирует существующее или следует из него
- **НЕ** добавляй правило-workaround — фикси причину, а не симптом
- **НЕ** записывай в CLAUDE.md без подтверждения пользователя — предложи правило, дождись ОК
- Ориентир: ~30-40 правил в корневом CLAUDE.md. Если больше — сначала ревизия, потом добавление
- Если правило актуально только для одного проекта — оно идёт в проектный CLAUDE.md, не в корневой

### Ревизия

- Перед добавлением правила — проверь, нет ли дубля или противоречия
- Если правило стало очевидным (усвоено на уровне привычки) — удали
- Если правило было реакцией на единичный случай и больше не актуально — удали
- При `/compact` в конце длинной сессии — предложи ревизию если видишь устаревшее

## Управление контекстом (предотвращение переполнения)

**Проблема:** при работе над несколькими задачами в одной сессии контекст переполняется → потеря прогресса.

### Команды управления контекстом

| Команда | Когда использовать |
|---------|-------------------|
| `/context` | Проверить сколько контекста занято прямо сейчас |
| `/compact` | Сжать историю в саммари (задача не закончена, но контекст разросся) |
| `/clear` | Полный сброс при переходе на несвязанную задачу |

**Проактивно предлагай `/compact`** когда контекст явно разросся (много файлов прочитано, длинная дискуссия). Не жди пока пользователь упрётся в лимит.

### Правила

1. **Максимум 2-3 задачи за сессию.** Если задач больше — предупреди пользователя и предложи разбить на несколько сессий.

2. **Делегируй в субагенты.** Исследование файлов, поиск по кодовой базе, чтение больших файлов — выноси в субагенты. Это не засоряет основной контекст.

3. **Читай файлы экономно:**
   - Не читай весь файл, если нужны только конкретные строки — используй `offset` + `limit`
   - Не читай файл повторно, если уже читал в этой сессии
   - Для больших файлов (>200 строк) сначала ищи нужное через `Grep`, потом читай конкретный фрагмент

4. **Пиши план в файл, а не в контекст.** Если нужен план работ — запиши его в файл, а не держи в голове.

5. **После завершения каждой задачи** — фиксируй результат (commit или запись в файл). Если сессия оборвётся, прогресс не потеряется.

6. **Если задач >3 — создай файл-план** со списком задач и статусами. Работай по нему, отмечая выполненное.

## LLM Wiki — паттерн для доменных знаний (опционально)

Если хочешь вести кросс-проектную базу знаний (статьи, обзоры инструментов, паттерны) — подними `wiki/` в одном из подпроектов по паттерну [Karpathy LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f).
